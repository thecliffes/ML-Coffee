ML Coffee - KNN Model Documentation
=====================================
Project: Coffee bean recommender using KNN classification
Target: Predict coffee bean name from tasting attributes
Model file: model.ipynb


FEATURES
--------
Input features: roast_level, aroma, structure, body, flavour, aftertaste
Target: name (specific coffee bean)


FEATURE ENCODING DECISIONS
---------------------------

1. roast_level — OrdinalEncoder
   Categories (in order): Light, Medium-Light, Medium, Medium-Dark, Dark
   Mapped to integers:     0     1             2      3           4

   Why ordinal over one-hot:
   - Roast level has a natural ordering — Light is genuinely "less roasted" than Dark
   - KNN uses Euclidean distance, so encoding the order numerically means distances
     between roast levels are meaningful (Light vs Medium-Light = 1, Light vs Dark = 4)
   - One-hot would make all roast levels appear equidistant, losing that relationship

   Safety setting: handle_unknown='use_encoded_value', unknown_value=-1
   - Prevents a crash if an unseen roast string appears at prediction time

2. aroma, structure, body, flavour, aftertaste — SimpleImputer (median)
   - These are already numeric, so no encoding needed
   - structure has significant missing values (not all reviews include acidity/structure)
   - strategy='median' fills NaN values with the column median from training data
   - Median preferred over mean: more robust to skewed distributions or outlier scores
   - Allows all rows to be used without dropping incomplete reviews

3. StandardScaler — added to both branches (decision made after initial setup)
   Why scaling matters for KNN:
   - KNN computes Euclidean distance, so features with larger numeric ranges dominate
   - Before scaling: roast_level was 0-4, numeric scores were ~0-10
   - This gave numeric features ~2.5x more influence on distance than roast_level
   - StandardScaler centres each feature at mean=0, std=1, putting all on equal footing

   Why StandardScaler over MinMaxScaler:
   - MinMaxScaler compresses to [0,1] but is sensitive to outliers — one extreme score
     shifts all other values in that feature
   - StandardScaler is more robust when scores are not perfectly uniformly distributed

   Implementation: chained as sub-pipelines inside the ColumnTransformer
   - Roast branch:   OrdinalEncoder -> StandardScaler
   - Numeric branch: SimpleImputer  -> StandardScaler


PREPROCESSOR STRUCTURE (final)
--------------------------------
ColumnTransformer(
    roast:   Pipeline([OrdinalEncoder(ordered), StandardScaler])  -> roast_level
    numeric: Pipeline([SimpleImputer(median),   StandardScaler])  -> aroma, structure, body, flavour, aftertaste
)


MODEL
------
Algorithm: K-Nearest Neighbours (KNeighborsClassifier)
Distance metric: Euclidean
K selection: CV sweep k=1 to 15, 5-fold cross-validation, pick best mean accuracy
Pipeline: preprocessor -> KNN


INITIAL MODEL RESULTS KNN
--------------------------
CV accuracy (exact match): 0.003 (0.3%)

Interpretation:
- This is expected and not a sign the model is broken
- The target has hundreds of unique coffee bean names
- Many beans share very similar tasting profiles, making exact match nearly impossible
- With ~300 unique classes, random guessing gives ~0.33% — so 0.003 is effectively at chance
  for exact match
- The model is designed as a recommender, not a single-answer classifier

Best k selected by CV: 8

Prediction confidence — 12.5% per top result:
- In KNN, probability = fraction of k neighbours that voted for that class
- 12.5% = 1/8 neighbours, which is the minimum possible vote share at k=8
- The top predictions each received only 1 vote out of 8 neighbours
- This indicates high uncertainty: the 8 nearest neighbours in training data were all
  different coffees, meaning many beans cluster closely in feature space
- Higher confidence (e.g. 37.5% = 3/8 votes) would indicate a clearer nearest-neighbour match

More meaningful metrics: top-N accuracy (right answer appears in top 3 or top 5 shortlist)

NOTES
------
